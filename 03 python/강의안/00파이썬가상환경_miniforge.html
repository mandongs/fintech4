Miniforge 중심 · 실습형

Miniforge(Conda) 강의안 — 가상환경 만들기, 패키지 설치, 환경 옮기기

이 문서는 Miniforge(conda-forge 중심) 기준으로 가상환경의 개념부터 conda·mamba·pip 설치 전략, 그리고 환경 재현/이식까지 실무 흐름으로 정리한 강의안이다.

가상환경이란 왜 필요한가 Miniforge란 핵심 개념 conda·mamba·pip 차이 설치/초기 설정 가상환경 만들기/관리 패키지 설치 전략 채널/우선순위 환경 옮기기/재현 pip 혼합 운영 문제 해결 실습 치트시트

가상환경이란

가상환경(virtual environment)은 한 컴퓨터(한 건물) 안에서 프로젝트마다 전용 주방을 따로 만들어 쓰는 “독립 공간”이다. 프로젝트별로 조리 방식(파이썬 버전)과 재료/양념(패키지), 필요하면 설비와 조리도구(바이너리 라이브러리)까지 따로 갖춘다.

요리 비유 핵심 프로젝트 A는 중화요리를 하는 주방, 프로젝트 B는 이탈리안 화덕피자를 만드는 주방이라고 생각하면 된다. 서로 조리 방식과 장비가 너무 다르기 때문에, 주방을 분리하지 않으면 충돌이 난다.
프로젝트 A: 중화요리 주방(예시)
  • 센 불 가스레인지 1구 (조리 방식/파이썬 버전)
  • 웍(Wok), 중식 칼, 튀김 온도계 (조리도구/바이너리)
  • 간장, 굴소스, 두반장 (패키지/의존성)
  • 빠른 볶음/고온 단시간 조리 (특정 연산/런타임 조합)
프로젝트 B: 이탈리안 화덕피자 주방(예시)
  • 장작 화덕 + 작업대, 가스레인지 3구 (조리 방식/파이썬 버전)
  • 피자 삽, 반죽 숙성 냉장고, 화덕 온도계 (조리도구/바이너리)
  • 올리브 오일, 모차렐라, 바질 (패키지/의존성)
  • 고온 장시간 굽기/숙성 (GPU, CUDA 같은 장비 의존 가능)
가상환경이 없을 때 주방이 하나뿐이면 “중식의 센 불 + 웍”과 “피자의 화덕 + 숙성”이 한 공간에 섞여 재료 냄새가 배고, 온도/도구/레시피가 충돌하며, 어느 한쪽이 자주 깨진다(오류/충돌).
핵심 가상환경은 단순히 “패키지 폴더를 분리”하는 수준을 넘어, 도구에 따라 파이썬 자체와 바이너리 설비(C/CUDA 등)까지 포함해 “주방 한 세트”를 통째로 분리한다.

왜 가상환경이 필요한가

1) 충돌을 막기 위함이다

하나의 주방에서 중식과 화덕피자를 같이 하면, 중식용 강한 향(소스/기름)과 화덕 온도, 조리 순서가 서로 간섭한다. 마찬가지로 전역 파이썬에 프로젝트별 요구사항을 섞으면, 한쪽을 맞추면 다른 한쪽이 깨지는 일이 생긴다.

  • 프로젝트마다 필요한 패키지 버전이 다를 수 있다.
  • 특정 라이브러리는 특정 파이썬 버전을 요구할 수 있다.

2) 재현성을 위해서이다

“오늘은 맛있던 요리”가 “내일은 맛이 달라지는” 이유는 보통 재료/설비가 바뀌어서이다. 가상환경을 파일로 기록하면, 팀원/서버에서도 동일한 “주방 세트”를 재현할 수 있다.

  • 개발 PC → 서버
  • 팀원 A → 팀원 B

3) 시스템(전역 환경)을 깨지 않기 위함이다

주방 하나에 모든 도구를 덕지덕지 붙이면 결국 동선도 망가지고, 고장도 잦다. 전역 파이썬에 설치를 쌓으면 OS 업데이트/권한 문제/패키지 충돌이 증가한다. 가상환경은 “시스템은 깨끗하게, 프로젝트는 분리” 원칙을 가능하게 한다.

4) 실험을 안전하게 분기하기 위함이다

같은 중식 레시피라도 새로운 소스 조합을 실험하고 싶으면 “시험 주방”이 필요하다. 환경을 복제(clone)하면 기존 환경을 보존한 채 실험을 분리할 수 있다. 

# 기존 환경을 복제해서 실험용 환경을 만든다.
conda create -n exp --clone myproj

Miniforge란

Miniforge는 conda 배포판 중 하나이며, 기본 채널이 conda-forge로 설정된 것이 핵심이다. 즉 “conda를 conda-forge 중심으로 쓰는 표준 구성”에 가깝다.

구분기본 채널성격추천 상황
Miniforge conda-forge 가볍고 커뮤니티 패키지 풍부 실무에서 가장 무난한 기본값
Miniconda defaults 가볍고 기본 conda 구성 defaults 중심 정책을 쓰는 조직
Anaconda defaults 대규모 패키지 포함 교육/오프라인 환경 등
결론 이 강의안은 Miniforge + conda-forge + strict를 기본 운영 방식으로 사용한다.

핵심 개념

환경(environment)

환경은 “파이썬 + 패키지 + (필요 시) 시스템 라이브러리” 묶음이다. 요리 비유로 보면, 환경은 “주방 한 세트”이다. 중식 주방에는 웍과 센 불이, 화덕피자 주방에는 화덕과 반죽 숙성 설비가 들어간다.

  • 파이썬 버전도 환경마다 다르게 설치할 수 있다.
  • MKL/OpenBLAS 같은 수치연산 라이브러리도 포함될 수 있다.

채널(channel)

채널은 패키지 저장소이다. 같은 “재료(패키지)”라도 공급처(채널)마다 규격/품질(빌드)이 달라 충돌이 날 수 있다. Miniforge는 conda-forge를 기본 공급처로 본다.

  • 채널 혼용을 줄이면 재현성이 좋아진다.
  • strict 우선순위는 “한 공급처 기준으로 맞춘다”에 가깝다.

conda · mamba · pip 설치 방식 차이

한 문장 요약

  • conda는 “주방을 만들고(환경) + 재료/도구를 한꺼번에 조합(해결)해서” 설치하는 관리자이다.
  • mamba는 conda와 거의 같은데 “장보기(의존성 계산)가 훨씬 빠른” 대체 도구이다.
  • pip는 “파이썬 재료(PyPI)” 중심으로 빠르게 설치하지만, 설비(바이너리)는 OS/휠에 많이 의존한다.

1) conda 설치의 특징

  • 환경을 만들고 그 안에 설치한다.
  • 패키지뿐 아니라 파이썬 자체도 conda로 설치한다.
  • 많은 패키지가 미리 빌드된 바이너리로 제공되어 설치가 안정적이다.
  • 대신 “전체 레시피 조합을 계산”하는 과정(solver)이 느릴 수 있다.
# conda-forge에서 numpy 설치 예시이다.
conda install -c conda-forge numpy

2) mamba 설치의 특징

  • 명령 형태가 conda와 매우 유사하다.
  • 의존성 해결(장보기 계산)이 빠르며, 큰 환경에서 체감이 크다.
  • 실무에서는 “conda 대신 mamba로 install/create”를 많이 사용한다.
# mamba로 설치 예시이다.
mamba install -c conda-forge numpy
기억 결과(설치되는 패키지)는 비슷하고, 체감 차이는 주로 “속도”에서 온다.

3) pip 설치의 특징

  • 기본 저장소는 PyPI이다.
  • 순수 파이썬 패키지는 매우 빠르고 편하다.
  • 하지만 C/CUDA 의존성이 있으면 “오븐 규격이 안 맞는 문제”처럼 OS/휠 호환 문제가 생길 수 있다.
  • 가상환경(venv/conda 환경 등) 안에서 사용해야 안전하다.
# conda 환경 안에서도 pip는 사용 가능하다.
conda activate myproj
python -m pip install fastapi uvicorn

언제 무엇을 쓰는가

상황추천이유
데이터/과학 + 바이너리 설비 복잡(웍/화덕급) conda 또는 mamba 설비(바이너리) 포함 조합이 안정적이다.
설치가 너무 느리다 mamba conda보다 solver(장보기 계산)가 빠르다.
conda-forge에 없는 패키지 pip PyPI에만 있는 경우가 많다.
순수 파이썬 웹/유틸 위주(도구 단순) pip(venv) 또는 conda 둘 다 가능하나 팀 표준에 맞추면 된다.

의존성 모델 차이(왜 충돌이 생기는가)

요리로 치면 “레시피 전체 조합을 먼저 점검하느냐” vs “재료를 일단 넣고 보느냐” 차이이다.

conda/mamba (주방 전체를 맞춘다)
  • 설치 전에 “전체 조합”을 계산한다.
  • 파이썬뿐 아니라 C 라이브러리도 함께 조합한다.
  • 채널/빌드/플랫폼을 강하게 고려한다.
pip (재료 중심으로 맞춘다)
  • Python 패키지 중심으로 의존성을 맞춘다.
  • 설비(바이너리)는 휠/시스템 환경에 의존한다.
  • 가끔 “설치는 됐는데 조리(실행) 중에 깨짐”이 발생한다.
중요 conda 환경에서 pip 설치를 많이 섞으면, conda가 관리하던 “설비 조합”이 깨질 수 있다. 기본 전략은 conda(또는 mamba)로 최대한 설치하고, 꼭 필요한 것만 pip로 추가하는 방식이다.

Miniforge 설치 후 초기 설정

설치 확인

conda --version
conda info
conda config --show channels

권장 설정

설정의미명령
auto_activate_base 터미널 시작 시 base 자동 활성화를 끈다. conda config --set auto_activate_base false
channel_priority 채널 혼용 충돌을 줄이기 위해 strict로 설정한다. conda config --set channel_priority strict
show_channel_urls 설치 출처를 표시한다. conda config --set show_channel_urls true 
# 설정 확인이다.
conda config --show

mamba 설치(선택이지만 추천)

Miniforge는 conda-forge 중심이므로, mamba도 conda-forge에서 설치하는 편이 자연스럽다. 

conda install -c conda-forge mamba

가상환경 만들기/관리

환경 생성

# conda로 생성이다.
conda create -n myproj python=3.11

# mamba로 생성이다(빠를 수 있다).
mamba create -n myproj python=3.11

활성화/비활성화

conda activate myproj
conda deactivate

환경 목록/삭제

conda env list
conda remove -n myproj --all

현재 python이 어느 환경인지 확인

python -V
python -c "import sys; print(sys.executable)"
python -m pip --version

패키지 설치 전략(실무 규칙)

규칙 1) conda-forge 중심으로 설치한다

Miniforge의 장점은 conda-forge 생태계에 있다. 가능하면 한 환경에서는 conda-forge만 쓰는 편이 안정적이다. 요리로 치면 “한 주방은 한 레시피 체계(한 공급처)”로 맞추는 것과 비슷하다. 

conda activate myproj
mamba install -c conda-forge numpy pandas matplotlib

규칙 2) pip는 “마지막 수단”처럼 사용한다

conda-forge에 없는 패키지, 또는 conda 패키지가 너무 오래된 경우에만 pip를 쓴다. 즉, 주방 기본 재료는 정규 공급처에서 받고, 특정 수입 재료만 별도로 구하는 느낌이다. 

conda activate myproj
python -m pip install some-package-only-on-pypi
중요 pip로 설치한 바이너리 패키지가 conda의 “설비 조합”을 깨면, 해결이 어려워져서 “새 주방(새 환경) 만들기”가 더 빠를 수 있다.

규칙 3) “환경을 작게” 유지한다

  • 주방에 도구가 많아질수록 정리/관리 비용이 커진다.
  • 환경이 커질수록 solver 시간과 충돌 가능성이 증가한다.
  • 실험용은 별도 환경으로 분리하는 편이 좋다.

채널(channel)과 우선순위

채널 확인

conda config --show channels
conda config --show channel_priority

채널 혼용이 위험한 이유

  • 같은 재료(패키지)라도 공급처(채널)마다 규격/품질(빌드)이 달라질 수 있다.
  • 결과적으로 의존성이 뒤집히며, 환경이 크게 변경될 수 있다.
  • 재현성(팀/서버 동일 환경)이 떨어진다.

가상환경 그대로 옮기기(재현/이식)

1) environment.yml로 재현(권장)

# 내보내기이다(이식성 높이기 위해 no-builds 권장).
conda env export -n myproj --no-builds > environment.yml

# 다른 PC/서버에서 재현이다.
conda env create -f environment.yml
from-history를 쓰는 경우이다

“직접 설치한 것만” 남기고 싶으면 아래가 유용하다. 다만 solver에 따라 결과가 달라질 수 있다. 

conda env export -n myproj --from-history > environment.yml

2) conda-pack으로 통째 옮기기(동일 OS/아키텍처)

conda install -c conda-forge conda-pack
conda pack -n myproj -o myproj.tar.gz
# 대상 머신에서 압축 해제 + 경로 보정이다.
mkdir -p ~/envs/myproj
tar -xzf myproj.tar.gz -C ~/envs/myproj
~/envs/myproj/bin/conda-unpack
주의 Windows↔Linux처럼 플랫폼이 다르면 conda-pack은 그대로 동작하지 않는다.

conda 환경에서 pip 혼합 운영

환경 파일에 pip 섹션 포함

name: myproj
channels:
  - conda-forge
dependencies:
  - python=3.11
  - numpy
  - pandas
  - pip
  - pip:
      - fastapi
      - uvicorn

pip 설치 시 체크리스트

  • 반드시 conda activate 후 설치하는 편이 좋다.
  • pip 대신 python -m pip를 쓰는 편이 좋다.
  • 바이너리 패키지 설치 후 실행 테스트를 바로 하는 편이 좋다.

자주 나는 문제 해결

solver가 느리거나 충돌이 잦을 때

  • 채널을 conda-forge 단일로 유지하는 편이 좋다.
  • channel_priority strict를 확인하는 편이 좋다.
  • conda 대신 mamba를 쓰면 속도가 개선될 수 있다.
conda config --show channel_priority

pip 설치 후 깨졌을 때

  1. 가능하면 pip로 설치한 패키지를 제거하는 단계이다.
  2. 안 되면 새 환경을 만들고 conda 기반으로 재구성하는 단계이다.
# 새 환경으로 재구성하는 방식이다.
mamba create -n clean python=3.11
mamba activate clean
mamba install -c conda-forge numpy pandas

치트시트

목표condamambapip
환경 생성 conda create -n X python=3.11 mamba create -n X python=3.11 해당 없음
패키지 설치 conda install pkg mamba install pkg python -m pip install pkg
환경 내보내기 conda env export -n X --no-builds > environment.yml
환경 재현 conda env create -f environment.yml
실무 기본 전략 Miniforge라면 conda-forge 단일 + strict를 기본으로 하고, 설치가 느리면 mamba로 create/install을 수행하고, conda-forge에 없을 때만 pip를 사용한다.

Miniforge 기반 conda 운영은 “채널 단일화 + strict + 환경을 작게 + yml로 기록”이 핵심이다.
