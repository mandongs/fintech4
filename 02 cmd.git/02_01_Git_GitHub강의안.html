1. Git과 GitHub 한 줄 요약

  • Git: 내 컴퓨터에서 파일 변경 이력을 관리하는 도구(버전 관리)
  • GitHub: Git 저장소를 인터넷에 올려 백업·공유·협업하는 서비스

비유

  • Git = “되돌리기 + 버전관리”
  • GitHub = “클라우드 협업 저장소”

2. 왜 Git을 써야 할까?

Git이 없으면:

  • final.zip → final_final.zip → final_final_진짜최종.zip …

Git을 쓰면:

  • 언제든 과거 상태로 되돌릴 수 있음
  • 누가/언제/무엇을 바꿨는지 기록이 남음
  • 실수해도 복구 가능

2-1. Git 설치 (Windows)

Git을 실제로 쓰려면 먼저 Windows에 Git for Windows를 설치해야 한다.

  • Git for Windows 최신 버전을 설치한다: https://git-scm.com
  • 설치 옵션은 기본값을 그대로 사용해도 된다.
  • 설치 후 CMD 또는 PowerShell을 새로 열어 아래 명령이 동작하는지 확인한다.
git --version
ssh -V
ssh-keygen -h
중요 설치 직후 기존에 열려 있던 터미널에서는 git이 인식되지 않을 수 있다. CMD/PowerShell을 닫고 다시 열어 확인하는 것이 안전하다.

3. Git 핵심 개념 (이것만 알면 됨)

  • Repository(저장소): 프로젝트 폴더 + 변경 이력
  • Commit(커밋): 변경 상태를 “사진 찍듯이” 저장한 기록
  • Branch(브랜치): 작업용 분기(기능 개발/실험용)

기본 브랜치 이름: main

3-1. Git 최초 사용자 정보 설정 (반드시 1회 필요)

Git을 처음 설치한 뒤 git commit을 실행하면 아래와 같은 오류가 발생할 수 있다. 

Author identity unknown
fatal: unable to auto-detect email address

Git은 모든 커밋에 작성자 이름(name)과 이메일(email) 정보가 반드시 필요하다. 이는 오류가 아니라 정상적인 초기 설정 단계이다.

전역 설정 (권장, 최초 1회)

아래 설정은 한 번만 하면 이후 모든 저장소에서 공통으로 적용된다.

git config --global user.name "홍길동"
git config --global user.email "hong@example.com"
참고 이메일은 GitHub 계정 이메일과 같을 필요는 없지만, 같을 경우 GitHub 커밋 기록(잔디)이 정상적으로 연결된다.

설정 확인

git config --global --list

저장소별 설정 (선택)

회사 프로젝트나 실습용 저장소 등, 특정 저장소에서만 다른 정보를 쓰고 싶을 때 사용한다. 

git config user.name "Lab User"
git config user.email "lab@example.com"
정리 Git은 처음 사용할 때 반드시 사용자 정보 설정이 필요하며, 이 설정은 PC당 1회만 수행하면 된다.

파일 수정

↓

  • git commit : “이 상태를 기록으로 남길게”

6. GitHub는 왜 필요할까?

  • 내 코드 백업
  • 여러 사람이 함께 하는 협업
  • 포트폴리오 공개/배포

Git은 로컬에서도 되지만, GitHub는 원격(온라인) 저장소입니다.

7. GitHub를 SSH로 연결하기 (Windows CMD/PowerShell 기준)

7-1. 준비물

  • Git 설치 및 설치 확인(2-1 절)이 완료된 상태이다.
  • Windows CMD 또는 PowerShell을 사용할 수 있어야 한다.
  • GitHub 계정이 있어야 한다.
정리 이 절에서는 “SSH로 GitHub에 연결하는 방법”만 다룬다.

7-2. SSH 키 생성 (ed25519만 사용)

CMD/PowerShell에서 실행:

ssh-keygen -t ed25519 -C "your_email@example.com"

ed25519 의미 (초보자용)

  • -t = 키 타입(암호화 방식)
  • ed25519 = 최신 표준(빠르고 안전하고 키가 짧음)
    → 이 강의안에서는 ed25519만 사용합니다.

7-3. 공개키 확인 (Windows CMD/PowerShell)

아래 명령으로 공개키를 출력해서 전체를 복사합니다.

type %USERPROFILE%\.
ssh\id_ed25519.pub

7-4. GitHub에 SSH 키 등록

  1. GitHub → Settings
  2. SSH and GPG keys
  3. New SSH key
  4. Title: 내 PC 이름
  5. Key: 방금 복사한 공개키 붙여넣기

7-5. SSH 연결 테스트 (Windows)

처음 연결이면 yes 입력이 나올 수 있습니다.

ssh -T git@github.com

9. 100MB 이상 파일 때문에 push 실패할 때 (Windows)

GitHub는 단일 파일 100MB 초과 업로드를 제한합니다.

9-1. 가장 좋은 해결: .gitignore에 미리 제외

예시:

gitignore
# 대용량/바이너리
*.zip
*.tar
*.tar.gz
*.mp4
*.bin
*.pt
*.ckpt
*.h5
# 데이터/모델 폴더
data/
logs/
models/

9-2. 이미 커밋에 포함해버렸다면 (핵심)

.gitignore만 추가해도 이미 커밋된 파일은 계속 추적됩니다.
따라서 “추적에서 제거”가 필요합니다.

  1. Git 추적에서 제거:

9-3. 대용량 파일이 꼭 필요하면: Git LFS

(조직/프로젝트 정책에 따라 사용)

git lfs install
git lfs track "*.pt"
git add.
.gitattributes
git commit -m "Track large files with LFS"
git push

10. git reset vs git revert 차이 (초보자 필수)

한 줄 결론

  • reset: 내 로컬 히스토리를 되감음(기록이 바뀜) → push 전에만 사용 권장
  • revert: 되돌리는 내용을 담은 새 커밋을 추가(기록 보존) → push 후/협업에서 안전

10-1. reset (push 전 로컬 정리용)

git reset --soft HEAD~1
git reset --hard HEAD~1

10-2. revert (push 후/협업 브랜치에서 안전)

git revert 커밋해시

11. 자주 하는 실수 (Windows 기준)

  • add 안 하고 commit
  • 커밋 메시지 없이 작업
  • pull 안 하고 push
  • 대용량 파일을 그대로 커밋

✅ 해결 습관: 작게 자주 commit, push 전에 git status
